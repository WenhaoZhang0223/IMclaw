# IMclaw workfile 文件库检索与发送设计

## 目标

将项目根目录现有的 `workfile/` 作为用户自行维护的长期文件库。用户在飞书私聊中用自然语言要求“把某个文件发给我”时，IMclaw 递归检索文件库；明确唯一匹配时直接发送，存在歧义时先列出候选并等待用户确认。

`workfile/` 是只读文件库。IMclaw 不创建、修改、重命名、移动或删除其中的内容，文件发送成功后也必须保留原件。

## 非目标

- 不接收或保存用户从飞书上传的文件。
- 不读取 PDF、Word、Excel 等文件的正文来建立全文索引。
- 不建立持久化索引、数据库或文件系统监听器。
- 不在 `workfile/` 之外查找用户请求的已有文件。
- 不修改现有 `deliver_file` 的“发送生成文件后删除”语义。
- 不修改 pi-mono 的核心 Agent、AI 或 Coding Agent 包。

## 目录约定

文件库固定为：

```text
IMCLAW_WORKSPACE/workfile
```

当前部署中对应项目内已有的 `workfile/` 目录。系统每次检索时实时扫描该目录，因此用户新增、替换或整理文件后无需重建索引。

目录不存在、不是目录或无法读取时返回明确的配置错误。系统不得自动创建目录。

## 架构

### `find_workfile(query)`

新增只读检索工具并注册到每个 IMclaw Agent Session：

```text
find_workfile(query)
```

工具递归扫描 `workfile/`，只收集普通文件，并根据以下元数据对自然语言查询进行排序：

- 文件名；
- 不含扩展名的主文件名；
- `workfile/` 下的相对目录；
- 扩展名；
- 文件修改时间。

基础匹配忽略大小写，并弱化空格、横线和下划线差异。工具不读取文件正文。结果最多返回 10 个候选，每项包含：

- `workfile/` 下的相对路径；
- 文件类型；
- 文件大小；
- 修改时间；
- 匹配分数。

工具返回结构化候选，由 Agent 结合用户原始自然语言和候选分数判断是否存在明确匹配。

### `send_workfile(path)`

新增只读发送工具：

```text
send_workfile(path)
```

参数只能是 `workfile/` 下、由检索工具返回的相对路径。工具绑定当前会话的 `chatId`，复用现有飞书文件上传和文件消息发送能力。Agent 无法指定其他接收人。

该工具与 `deliver_file` 保持独立：

- `deliver_file` 用于 Agent 新生成的交付文件，发送成功后删除本地文件；
- `send_workfile` 用于用户长期保存的已有文件，发送成功后保留原文件。

## 交互规则

用户提出文件请求后，Agent 必须先调用 `find_workfile`：

1. 候选明确唯一、查询明确命中文件名，或第一名明显优于其他结果时，直接调用 `send_workfile`。
2. 多个结果相近或置信度不足时，不发送文件；在飞书中列出带编号的候选文件名和相对路径，请用户确认。
3. 用户通过“第 2 个”、文件名或相对路径确认后，Agent 调用 `send_workfile`。
4. 没有匹配时明确告知未找到，并建议补充文件名、目录、类型或时间信息。
5. 候选超过返回上限时提示用户补充关键词，不默认为其发送第一项。

飞书回复不得暴露完整 Windows 绝对路径。

## 数据流

```text
用户用自然语言请求已有文件
  -> Agent 调用 find_workfile(query)
  -> 实时递归扫描 workfile/
  -> 返回按相关度排序的结构化候选
  -> 明确匹配：Agent 调用 send_workfile(relativePath)
  -> 歧义匹配：Agent 列出候选并等待用户确认
  -> send_workfile 再次校验文件真实路径和状态
  -> FeishuAdapter 上传文件并发送到当前 chatId
  -> 保留 workfile/ 中的原文件
  -> Agent 发送简短确认
```

## 安全与校验

两个工具都以 `IMCLAW_WORKSPACE/workfile` 的真实路径作为唯一允许边界。

### 检索校验

- 不跟随逃逸到文件库之外的符号链接。
- 只返回真实路径仍位于文件库内的普通文件。
- 无法读取的单个目录或文件不会扩大访问范围；工具记录安全日志并继续处理其他可读条目。
- 返回给 Agent 和用户的路径始终是文件库相对路径。

### 发送校验

在任何网络操作前：

1. 只接受非空相对路径，拒绝绝对路径。
2. 基于文件库解析路径并执行真实路径解析。
3. 拒绝 `..`、符号链接或其他形式的边界逃逸。
4. 确认目标仍然存在且是普通文件。
5. 确认文件非空且不超过 30 MB。
6. 使用目标文件的 basename 作为飞书显示名称。

即使文件曾由 `find_workfile` 返回，`send_workfile` 仍独立执行全部校验，以处理候选列出后文件被移动、替换或删除的情况。

## 匹配策略

检索器采用确定性的元数据评分，不调用模型、不读取文件内容：

- 完整文件名或完整相对路径命中获得最高权重；
- 主文件名中的查询词覆盖率次之；
- 目录名和扩展名命中提供附加权重；
- “最新”“最近”等时间意图使用修改时间调整排序；
- 普通自然语言礼貌词和发送意图词不应主导相关度。

“高置信度”由 Agent 根据查询与候选共同判断，而不是由检索器自动发送。系统提示明确要求：只有唯一强匹配或第一名明显领先时才可直接发送；任何合理歧义都必须请求用户确认。

## 错误处理

- 文件库不存在或不可读：返回文件库配置错误。
- 未找到候选：返回空结果和可操作提示。
- 扫描中个别条目不可读：记录路径和错误，继续扫描其余条目。
- 文件在确认后消失或不再是普通文件：停止发送并提示用户重新检索。
- 文件为空或超过 30 MB：拒绝发送并说明原因。
- 飞书上传失败：保留原文件并返回上传失败。
- 飞书消息发送失败：保留原文件并返回消息发送失败。
- 发送成功：保留原文件，不自动重试。

日志不得记录文件内容、App Secret、OAuth 令牌或其他凭证。

## 系统提示

IMclaw 身份提示增加以下规则：

- `workfile/` 是用户维护的只读长期文件库。
- 请求已有文件时先调用 `find_workfile`，不得用通用 shell 工具绕过检索边界。
- 明确匹配可直接发送；相近候选必须先请用户确认。
- 只能把检索结果中的相对路径传给 `send_workfile`。
- 不得修改、移动或删除 `workfile/` 中的任何内容。
- 成功调用 `send_workfile` 表示文件已发送，不要让用户自行查找本地路径。

## 测试设计

### 检索工具

- 递归发现嵌套子目录中的普通文件。
- 中英文文件名均可检索。
- 文件名、目录名、扩展名和修改时间参与排序。
- 大小写、空格、横线和下划线差异不会破坏基础匹配。
- “最新”等时间意图按修改时间调整结果。
- 最多返回 10 个结构化候选。
- 不读取文件正文。
- 文件库缺失或不可读时返回配置错误。
- 符号链接不能使结果逃逸到文件库之外。

### 发送工具

- 能发送嵌套目录中的普通非空文件。
- 发送成功后原文件仍然存在且内容未变。
- 拒绝绝对路径、`..` 逃逸和符号链接逃逸。
- 拒绝不存在的文件、目录、空文件和超过 30 MB 的文件。
- 候选返回后文件被移动或删除时安全失败。
- 上传或消息发送失败时原文件保持不变。
- 始终绑定当前 `chatId`。

### Pi 集成

- `find_workfile` 和 `send_workfile` 注册到每个 IMclaw Session。
- 系统提示包含只读、先检索、歧义确认和相对路径规则。
- 候选确认可依靠现有持久化聊天上下文完成，不新增服务器端候选状态。
- 现有 `deliver_file` 的发送后删除行为保持不变。
- 现有 `/help`、`/new`、`/status`、`/abort`、FIFO 队列和任务守卫行为保持不变。

测试使用临时目录和虚假文件发送器，不调用真实飞书、模型或浏览器。实现完成后运行新增的 IMclaw 定向测试、IMclaw 相关测试和仓库规定的 `npm run check`。

## 验收标准

1. 用户自行把不同格式的文件放入项目的 `workfile/` 及其子目录。
2. 用户在飞书用自然语言请求其中一个文件。
3. 唯一强匹配时，IMclaw 无需额外确认便发送正确文件。
4. 多个相近候选时，IMclaw 列出候选并等待用户选择，不擅自发送。
5. 用户确认候选后，飞书收到可下载的文件消息。
6. 发送完成、上传失败或消息发送失败后，`workfile/` 中的原文件均未被删除或修改。
7. IMclaw 无法通过这两个工具读取或发送 `workfile/` 外的文件。
8. 类型检查、格式检查和相关测试全部通过。
