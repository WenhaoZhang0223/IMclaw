# IMclaw 文件回传与发送后清理设计

## 目标

为 IMclaw 增加显式的 `deliver_file(path)` Agent 工具。Agent 在工作目录中生成用户需要的文件后，调用该工具将文件上传并发送到当前飞书私聊。只有当文件消息发送成功后，系统才删除实际回传的本地文件。

本功能支持工作目录内任意常见非空文件，单个文件最大 30 MB。它不扫描目录、不自动推测需要发送的文件，也不删除其他生成文件或未纳入 Git 的文件。

## 非目标

- 不接收用户从飞书上传的文件。
- 不自动发送任务期间出现的所有新文件。
- 不递归打包目录。
- 不修改 pi-mono 的核心 Agent、AI 或 Coding Agent 包。
- 不实现断点续传、自动重试或飞书云端文件清理。

## 架构

### `deliver_file` 自定义工具

新增独立的文件回传工具定义，并通过 pi-coding-agent 的 `customTools` 注册到每个 IMclaw Agent Session。工具参数只有文件路径：

```text
deliver_file(path)
```

工具绑定当前会话的 `chatId`、`IMCLAW_WORKSPACE` 和文件发送服务。Agent 不能指定目标聊天，防止把文件发送到其他会话。

### `ImAdapter` 扩展

`ImAdapter` 新增文件发送能力：

```text
sendFile(chatId, filePath, fileName)
```

`FeishuAdapter` 使用飞书 SDK 完成两步操作：

1. 调用 `im.v1.file.create` 上传文件并获得 `file_key`。
2. 调用 `im.v1.message.create`，以 `file` 消息类型把 `file_key` 发送到当前 `chatId`。

飞书 SDK 支持的明确类型为 `pdf`、`doc`、`xls`、`ppt`、`mp4`、`opus` 和 `stream`。系统按扩展名映射已知类型，其余常见文件使用 `stream`。

### Pi 集成

`PiAgentBackend` 在创建每个 chat runtime 时构造对应的 `deliver_file` 工具，并通过 `customTools` 传入 Session。系统提示增加以下规则：

- 用户要求下载或回传文件时，将最终文件写到工作目录内。
- 文件完成后调用 `deliver_file`。
- 不把仅供内部使用的中间文件回传。
- 工具成功即表示文件已经发送，无需在文本中声称仍需用户自行查找本地路径。

## 数据流

```text
用户请求文件
  -> Agent 使用现有 write/bash 工具生成工作目录内的文件
  -> Agent 调用 deliver_file(path)
  -> 校验路径、文件类型和大小
  -> FeishuAdapter 上传文件并取得 file_key
  -> FeishuAdapter 向当前 chatId 发送文件消息
  -> 删除刚刚发送的本地文件
  -> 工具返回成功结果
  -> Agent 发送简短文本确认
```

## 安全与校验

`deliver_file` 在任何网络操作之前执行以下校验：

1. 将相对路径基于 `IMCLAW_WORKSPACE` 解析。
2. 对工作目录和目标文件执行真实路径解析。
3. 确认目标真实路径位于工作目录真实路径内，拒绝 `..` 和符号链接逃逸。
4. 确认目标存在且是普通文件，不允许目录。
5. 确认文件非空且不超过 30 MB。
6. 使用目标文件的 basename 作为飞书显示文件名，不允许 Agent 单独伪造显示名称。

工具不会读取或发送工作目录外的文件，包括 Pi 认证文件、环境配置和用户主目录中的其他内容。

## 清理语义

本地删除严格发生在飞书文件消息发送成功之后：

- 上传失败：保留文件。
- 上传成功但消息发送失败：保留文件。
- 上传和消息发送成功：删除实际发送的文件。
- 发送成功但删除失败：工具返回“已发送但本地清理失败”，记录错误并让 Agent 明确告知用户。
- 任务被 `/abort` 中止且尚未发送：保留文件。

不自动重试上传或发送，以避免产生重复文件消息。用户可以在失败后要求 Agent 再次调用同一文件路径。

## 错误处理

面向 Agent 的错误结果应简短且可操作，包括：

- 文件不存在。
- 文件不在工作目录内。
- 目标不是普通文件。
- 文件为空。
- 文件超过 30 MB。
- 飞书上传失败。
- 飞书消息发送失败。
- 文件已发送但本地删除失败。

日志保留 SDK 错误和内部上下文，但不记录文件内容、App Secret、OAuth 令牌或其他凭证。

## 飞书配置要求

应用必须保持机器人能力和“以应用身份发送消息”权限。文件上传接口或租户策略若要求新增资源权限，部署验收时依据飞书后台/API 返回的明确权限提示开通，并重新发布应用版本。

## 测试设计

### 文件回传工具单元测试

- 工作目录内普通文件能够发送，并在成功后删除。
- 相对路径和工作目录内绝对路径均正确解析。
- 工作目录外路径、`..` 逃逸和符号链接逃逸被拒绝。
- 目录、空文件和超过 30 MB 的文件被拒绝。
- 上传/发送服务抛错时文件保留。
- 发送成功但删除失败时返回部分成功结果。
- 仅删除传入工具且已成功发送的文件，其他文件保持不变。

### 飞书适配器测试

- 扩展名正确映射到飞书文件类型。
- 上传请求包含文件流、basename 和正确类型。
- 缺少 `file_key` 时失败，不发送消息。
- 文件消息使用 `msg_type: "file"` 和 `content: { file_key }`。
- 上传或消息接口失败时向调用方传播错误。

### Pi 与 Controller 回归测试

- `deliver_file` 注册到 IMclaw Session，并固定绑定当前 `chatId`。
- 现有 `/help`、`/new`、`/status`、`/abort`、文本分片和每聊天 FIFO 行为保持不变。
- 现有 IMclaw 类型检查和所有测试继续通过。

## 验收标准

1. 用户在飞书要求生成一个常见文件。
2. Agent 在 `IMCLAW_WORKSPACE` 内生成文件并调用 `deliver_file`。
3. 飞书聊天收到可下载的文件消息。
4. 文件发送成功后，本地原文件不存在。
5. 工作目录中的其他文件没有被删除。
6. 发送失败时，本地原文件仍然存在并可重试。
7. 类型检查、格式检查和 IMclaw 测试全部通过。
